# docuseal-go

Go SDK for the DocuSeal API. Hybrid architecture: models are generated from
the OpenAPI spec, the client is written by hand.

## Architecture

- `types.go` - generated by oapi-codegen (types only), never edit by hand. Regenerate with `./generate-types.sh [spec-path-or-url]`.
  Response types come from named OpenAPI components (`GetTemplateResponse`,
  `TemplateField`, ...), request bodies are `<OperationId>Request` components
  (`CreateSubmissionRequest`), query params are generated per operation
  (`GetTemplatesParams`). Nullable fields are plain pointers (`*string`).
  Webhook payload schemas are excluded from generation.
- `client.go` - hand-written core: `Client`, options (base URL for
  Global/EU/on-premises), `do()` with auth header and error handling,
  `queryValues()` params serialization.
- `api.go` - hand-written API methods, one per endpoint. Mirrors the method
  naming of docuseal-js (`listTemplates` -> `ListTemplates`).

## Source of truth

The OpenAPI spec is generated by `lib/api_spec/` in the docuseal-website
repository and served at https://console.docuseal.com/openapi.yml. Response
top-levels are `<OperationId>Response` components, shared nested objects are
domain-named components (`TemplateField`); both live in
`lib/api_spec/schemas.rb` + endpoint files. Do not fix type issues here -
fix them in the spec and regenerate.

## Conventions

- `CreateSubmission` posts to `/submissions/init` (newer response shape),
  matching docuseal-js. The legacy `POST /submissions` array response is not
  exposed.
- `PermanentlyDelete*` methods use the DELETE endpoints with
  `?permanently=true`.
- Nested request objects are named components in the spec
  (`CreateSubmissionRequestSubmitter`, ...), so requests are built with
  typed literals; `Ptr` fills optional pointer fields.
